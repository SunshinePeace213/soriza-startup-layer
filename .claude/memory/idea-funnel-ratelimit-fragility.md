The `/idea-funnel` engine (`.claude/workflows/idea-funnel-engine.js`) fans out ~60 web-search-heavy Opus agents in the disconfirmation/demand stage. The web-research-heavy `market-researcher` (demand-detection) and `disconfirmation-judge` agents intermittently **finish without calling StructuredOutput** (likely a natural-stop / rate-limit interaction — `claude-opus-4-8[1m] temporarily unavailable`). Bare `agent({schema})` then THROWS, the throw escapes parallel()/pipeline()'s per-thunk catch, nulls the candidate, and crashes the run before persist. Two confirmed instances, both 2026-06-05: "Company Brain" (85 agents/777k tokens wiped); "SaaS Challengers" (this session — demand flaked on 9/10, ERP "won by forfeit," run crashed at the harness `.filter`).

Third confirmed instance, 2026-06-05: the **"AI Operating System for Companies"** run (worktree `idea-funnel-ai-company-os`). First run crashed at the harness `.filter(c.killed_at)` on null candidates after 9/10 disconfirmation-judges flaked (root cause caught verbatim in a market-researcher transcript: `API Error: Server is temporarily limiting requests (not your usage limit) · Rate limited`). Hardened with `safeAgent` (degrade-not-throw on the judge + fit + hypothesis calls) + null-guarded counts, then resumed → completed, but only 4/10 demand agents had succeeded (6 rate-limited again). Fixed by the controlled-backfill tactic below; this **reshuffled the shortlist** — the resume's provisional top-2 (engineering-context, knowledge-worker) were displaced by memory-layer-api (80/87) and post-meeting-action-router (80/85) once the 6 were measured. Lesson reinforced: a rate-limit-degraded board hands the founder the WRONG survivor.

**Hardening (what works):** wrap every schema/kill-capable stage agent in a retry-on-throw-or-null wrapper that NEVER propagates — degrade the candidate ALIVE (kept, low-ranked, flagged), never null/kill it. Also: (a) fit/hypothesis must only kill on a REAL `drop`/`kill` verdict, never on a null result (infra ≠ fatal flaw); (b) a demand fallback — if market-researcher flakes, extract from the `market-research.md` it wrote, else default to thin-but-reachable (never `reachable=false`, which would false-trigger fatal-flaw ③); (c) wrap the whole disconfirm/demand stage in try/catch returning the candidate; (d) null-guards in persist/counts. Honors the "no silent kills / never empty" contract. (Wrapper named `safeAgent` in the Company-Brain fix, `agentSafe` here — see next point.)

**THE FIX KEEPS GETTING LOST.** Both hardenings were done inside a bg worktree and never merged to main, so main's engine reverts to the un-guarded version and the NEXT run re-crashes from scratch. **After hardening the engine, merge the engine file to main** (or it is wasted). This is the root recurring failure, more than the rate-limit itself.

**Why:** an infra failure must never crash the run or silently kill a candidate — indistinguishable from a real fatal-flaw verdict; it destroys the board's trustworthiness.

**How to run it reliably:**
- **Controlled hand-backfill beats a 3rd full resume for rate-limited demand agents.** When the resume completes but N demand (market-researcher) agents flaked, do NOT re-resume the whole engine (another ~1M-token burst that re-hits the same rate-limit storm). Instead spawn the missing `market-researcher` agents DIRECTLY via the Agent tool in small batches (2–3 concurrent — well under the rate-limit threshold; zero failures at that concurrency), each writing its own `market-research.md` and returning the ranking fields. Then regenerate the ledger by calling `ledger-writer` with a FRESH prompt carrying the authoritative final table (a changed prompt = cache MISS = a real re-write, sidestepping the stale-cache trap below). Confirmed cheap + reliable on the AI-Company-OS run.
- **Resume-after-edit works for AGENT cache, but file-writing agents can cache-hit and re-persist STALE files.** Confirmed 2026-06-05 (dynamic-software-interfaces run): after fix+resume the JS state was correct (`dropped_to_null:0`, 10 candidates, return `counts.cleared_desk:10`) but the **ledger-writer was a cache hit** — it returned its cached structured result WITHOUT re-executing its Write, so `ledger.md`/`ledger.json` on disk still showed only the **7** candidates from the pre-fix run. The return value lies about what's on disk. **Always verify the persisted ledger's candidate count against the return `counts` after a resume**; if stale, reconcile by hand or force the writer to re-run (change its prompt/opts so it's a cache miss). Same caveat for any file-writing agent whose inputs changed but whose call signature didn't.
- **Resume needs args.** `Workflow({scriptPath, resumeFromRunId})` WITHOUT re-passing `args` exits immediately (`Provide args.thesis...`). Always re-pass the original args object.
- Smaller **n** (n=6) reduces the concurrent Opus burst, but n=10 completed fine via resume + retries here.
- Each run globs a new `_funnel-runs/<thesis-slug>-NN` label; delete the stale partial folder before a resume so the writer regenerates `-01` cleanly instead of bumping to `-02`.

**Separate engine bug (also 2026-06-05):** `disconfirmation-judge` is prompted to write `${ns}disconfirmation-brief.md` but its agent definition has **no Write tool**, so the brief FILE is never written (content survives only in the judge's structured return + the Phase A run-pack appendix). Downstream `/customer-discovery` requires `disconfirmation-brief.md` — backfill it for the survivor. **Best source = the judge's StructuredOutput in the workflow transcripts** (`subagents/workflows/<wf_id>/agent-*.jsonl`; grep for the candidate slug + `open_assumptions`) — it carries the full authoritative brief (all 9 ranked_risks/open_assumptions/interview_questions), more complete than the run-pack appendix. Prefer the latest/most-complete judge run if multiple exist. Real fix: give `disconfirmation-judge` a Write tool, or add a tiny brief-writer step.
